# 原则 2：简洁优先

这些示例用于在“直接清晰且足够完整”仍不清楚时辅助理解。

## 用直接清晰且足够完整的方案解决问题

推荐做法：
- 先覆盖当前需求真实需要的路径。
- 保持代码直接可读，不提前加入未被要求的配置、扩展点或抽象层。
- 如果最小改动会保留错误设计、重复逻辑或半成品实现，就扩大到足以完整解决问题的范围。

判断方式：
- 这个方案是否完整解决了用户请求或问题根因？
- 是否为了少改几行而留下明显错误结构？
- 是否加入了当前没有真实需求的抽象、配置或兼容层？

## 一次性流程保持直接

推荐做法：
- 对固定流程，优先写成直接的步骤序列。
- 只在多个真实调用方需要共享逻辑时抽取函数。
- 只在真实存在多个策略时引入策略、工厂或适配器。

适用场景：
- 只有一个调用方。
- 输入输出形态固定。
- 用户没有要求可配置或可扩展。

## 简单 LLM 调用直接使用 SDK

推荐做法：
- 直接调用 SDK。
- 保留清晰的 messages、model、temperature 和返回值处理。
- 只有当流程确实需要动态规划、多工具选择或长期状态时，才引入 agent 框架。

判断方式：
- 如果只是一次固定 prompt 调用，直接使用 SDK。
- 如果只是两个固定步骤，直接顺序调用。
- 如果需要根据中间结果动态选择工具、循环推理或维护长期状态，再考虑 agent 框架。

为什么：
LLM 应用尤其容易过度框架化；简单路径应保持可读、可调试、可验证。
