# 原则 3：精确改动

这些示例用于在“完整解决问题但避免无关改动”仍不清楚时辅助理解。

## 修复 bug 时保持改动聚焦

推荐做法：
- 只修改与用户请求或问题根因相关的代码路径。
- 如果根因需要修改多个位置、补测试或调整局部结构，就把这些作为必要改动完成。
- 遵循现有风格，保持无关变量名、文件位置和格式不变。
- 如果发现无关问题，单独告知，不在本次补丁里处理。

判断方式：
- 这行改动是否能追溯到用户请求或问题根因？
- 这个重命名、移动文件或格式化是否是解决问题所必需？
- 如果把这行改动拿掉，问题是否仍能完整解决？

## 必要改动可以跨多个位置

推荐做法：
- 如果问题根因分布在多个调用方，就同步修改相关调用方。
- 如果行为变化需要测试覆盖，就补充或更新测试。
- 如果局部结构阻止问题被完整修复，就调整该局部结构。

为什么：
精确改动不是最小 diff。为了完整解决根因而修改多个相关位置，仍然是精确改动。

## 修改完成后清理

推荐做法：
- 移除因本次改动而不再使用的导入、变量、函数等。
- 只清理本次改动造成的孤立代码。
- 发现与本次改动无关、原本就存在的死代码时，提出来，不要擅自删除。

判断方式：
- 这个未使用项是否由本次改动造成？如果是，清理。
- 这个死代码是否原本就存在且与本次请求无关？如果是，说明即可。

为什么：
收尾清理应保证本次改动自洽，但不应把无关历史问题混进同一个补丁。
