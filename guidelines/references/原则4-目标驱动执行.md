# 原则 4：目标驱动执行

这些示例用于在“可验证完成目标”和“验证路径”仍不清楚时辅助理解。

## 不同任务的可验证完成目标

- “添加校验” → “无效输入会被测试覆盖，并且测试通过”
- “修复 bug” → “有测试或最小复现能证明 bug 已复现并被修复”
- “重构 X” → “重构前后的行为一致，并通过已有测试、构建或运行路径验证”
- “优化性能” → “指定场景的指标被记录，并能和修改前结果比较”
- “调整 UI” → “在目标页面实际操作黄金路径和关键边界场景”

## 现有验证手段不足时构建临时验证入口

推荐做法：
- 先尝试使用项目已有测试、构建或运行路径。
- 如果现有验证手段不足，构建临时测试入口、调试脚本、最小复现页面或一次性命令来验证问题。
- 验证完成后移除临时代码。
- 如果临时代码有长期价值，将它转化为正式测试或正式调试工具。
- 如果无法可靠验证，明确说明未验证项。

适用场景：
- 项目没有覆盖该路径的测试。
- bug 只在特定输入、页面状态或集成路径下出现。
- 需要观察运行时输出、网络请求、数据库状态或浏览器行为。

## 多步骤任务写出验证点

推荐做法：

1. 修改输入校验逻辑 → 验证：无效输入测试失败后再通过。
2. 更新调用方处理 → 验证：相关集成路径仍能通过。
3. 清理本次改动造成的孤立代码 → 验证：无未使用导入、变量或函数。

为什么：
每一步都绑定验证点，可以减少做到一半就凭感觉宣称完成的情况。

## 无法可靠验证时说明边界

推荐做法：
- 明确说明已经验证了什么。
- 明确说明没有验证什么。
- 说明无法验证的原因，例如缺少依赖、环境不可用、外部服务不可访问或没有测试入口。
- 不把未验证的结果描述为已经完成验证。

表达方式：
- “已运行类型检查和单元测试；未验证浏览器交互，因为当前环境无法启动页面。”
- “已用临时脚本复现该输入路径；未覆盖生产数据规模下的性能表现。”
