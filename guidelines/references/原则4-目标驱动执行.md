# 原则 4：目标驱动执行

这些示例用于在“可验证完成目标”和“验证依据”仍不清楚时辅助理解。

## 把任务转化为可验证完成目标

- “添加校验” → 完成目标：“无效输入被拒绝，错误信息符合预期” → 验证依据：“覆盖该输入的测试失败后再通过”
- “修复 bug” → 完成目标：“最小复现中的错误行为消失” → 验证依据：“复现用例先失败，修改后通过”
- “重构 X” → 完成目标：“对外行为与重构前一致” → 验证依据：“覆盖该行为的已有测试、运行路径或快照对比通过”
- “优化性能” → 完成目标：“指定场景下目标指标改善” → 验证依据：“记录修改前后同一场景的可比指标”
- “调整 UI” → 完成目标：“目标页面的黄金路径和关键边界场景表现正确” → 验证依据：“在页面中实际操作并观察结果”

## 验证依据必须覆盖完成目标

推荐做法：
- 每个完成目标都要有验证依据覆盖。
- 一个验证依据可以覆盖多个完成目标，但必须说明覆盖关系。
- 不把无关测试、通用构建或普通自我总结当作验证依据。
- 优先选择能直接证明目标成立的最小验证路径，而不是机械运行全部测试。

示例：

任务：“修复注册表单在空邮箱时崩溃，并保留正常注册流程。”

完成目标与验证依据：
1. 空邮箱不再崩溃 → 验证：提交空邮箱时页面显示校验错误。
2. 正常邮箱仍可注册 → 验证：使用有效邮箱完成注册黄金路径。

不充分的验证：
- “已运行 unrelated-utils.test.ts，通过。”这个测试没有覆盖注册表单。
- “已运行构建，通过。”构建只能说明代码能编译，不能证明崩溃被修复。

## 现有验证手段不足时构建临时验证入口

推荐做法：
- 先尝试使用项目已有测试、构建或运行路径。
- 如果现有验证手段不足以证明完成目标，构建临时测试入口、调试脚本、最小复现页面或一次性命令来验证问题。
- 验证完成后移除临时代码。
- 如果临时代码有长期价值，将它转化为正式测试或正式调试工具。
- 如果无法可靠验证，明确说明未验证项。

适用场景：
- 项目没有覆盖该路径的测试。
- bug 只在特定输入、页面状态或集成路径下出现。
- 需要观察运行时输出、网络请求、数据库状态或浏览器行为。

示例：

任务：“修复某个 CSV 输入导致解析器进入死循环。”

推荐验证路径：
1. 用该 CSV 构建最小复现脚本，确认修改前会卡住或超时。
2. 修改解析器。
3. 重新运行最小复现脚本，确认能在预期时间内返回。
4. 删除临时脚本，或把它改成正式回归测试。

## 复杂任务先拆目标

复杂任务不要直接拆操作步骤，应先拆成可验证目标，再围绕目标执行与验证。

示例：

任务：“重构注册流程，并修复空邮箱崩溃问题。”

1. 目标：空邮箱不再崩溃 → 执行：修改邮箱校验与错误处理 ↔ 验证：提交空邮箱时显示校验错误。
2. 目标：正常注册流程行为不变 → 执行：调整注册流程调用方 ↔ 验证：有效邮箱仍能完成注册。
3. 目标：重构后代码自洽 → 执行：清理本次改动造成的孤立代码 ↔ 验证：无未使用导入、变量或函数。

为什么：
先拆可验证目标，再让执行与验证围绕目标互相校正，可以减少偏离目标或凭感觉宣称完成。

## 无法可靠验证时说明边界

推荐做法：
- 明确说明已经验证了什么。
- 明确说明没有验证什么。
- 说明无法验证的原因，例如缺少依赖、环境不可用、外部服务不可访问或没有测试入口。
- 不把未验证的结果描述为已经完成验证。

表达方式：
- “已用覆盖该输入路径的测试证明无效输入会被拒绝；未验证浏览器交互，因为当前环境无法启动页面。”
- “已用临时脚本复现并验证该输入路径；未覆盖生产数据规模下的性能表现。”
