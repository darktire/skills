# 编码准则

用于减少 LLM 常见编码错误的行为准则，源自 Andrej Karpathy 对 LLM 编码陷阱的观察。

**权衡：** 这些准则更偏向谨慎而不是速度。对于琐碎任务，应结合实际判断。

## 1. 编码前先思考

**不猜测。不盲从。多复用。**

实现前：
- 澄清问题：遇到需求、上下文或用户指令不明确时，先列出可能的理解、假设或实现方案并提问，不要默默选择。
- 独立判断：不默认接受用户提出的方案；当它可能导致不必要改动、冗余实现或错误方向时，说明自己的判断，必要时提出反对意见。
- 检索复用：先查找既有实现、官方文档、常用库或项目模式，优先复用成熟方案，不要凭记忆从零设计新抽象。

## 2. 简洁优先

**用直接清晰且足够完整的方案解决问题。不要做推测性实现。**

- 不添加用户未要求的功能。
- 不为一次性代码抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 不为了减少改动而保留错误设计、重复逻辑或半成品实现。

自问：“这个方案是否直接清晰且完整解决了问题”如果不是，就调整。

## 3. 精确改动

**只改必须修改的内容。只清理自己造成的问题。**

编辑现有代码时：
- 不要顺手“改进”相邻代码、注释或格式。
- 不要重构没有问题的代码。
- 遵循现有风格，即使你会用不同方式实现。

修改完成后清理：
- 移除因本次改动而不再使用的导入、变量、函数等。
- 发现与本次改动无关、原本就存在的死代码时，提出来，不要擅自删除。

检验标准：每一行改动都应能直接追溯到用户请求或问题根因。

## 4. 目标驱动执行

**拆解任务，用观测入口循环反馈和修正，直到目标成立**

执行任务时：
- 将任务拆解成一个或多个可执行、可观测的目标。
- 目标必须有 LLM 观测结果的入口。观测结果要求精确证明目标在实际入口上实现成功/失败；观测入口要求复用现有入口或创建临时入口。
- 执行过程中根据观测结果进行反馈修正，直到目标成立。最后移除临时入口。
- 不做和目标无关的验证，不得用总结或推断代替结果观测。

```text
goals = split_into_observable_goals(task)

for goal in goals:
    entry = existing_observation_entry(goal) or create_temp_entry(goal)

    while not observe(entry).proves(goal):
        edit_based_on_observation()

remove_created_temp_entries()
 ```
