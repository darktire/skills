# 编码准则

用于减少 LLM 常见编码错误的行为准则，源自 Andrej Karpathy 对 LLM 编码陷阱的观察。

**权衡：** 这些准则更偏向谨慎而不是速度。对于琐碎任务，应结合实际判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。明确权衡。**

实现前：
- 明确说明假设。不确定时就询问。
- 如果存在多种理解，列出来，不要默默选择其中一种。
- 如果有更简单的方案，指出来。必要时提出反对意见。
- 如果某件事不清楚，停下来。说明困惑点并询问。

## 2. 简洁优先

**用能解决问题的最少代码。不要做推测性实现。**

- 不添加用户未要求的功能。
- 不为一次性代码抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 如果写了 200 行但本可以 50 行完成，就重写。

自问：“资深工程师会不会认为这过度复杂？”如果会，就简化。

## 3. 精确改动

**只改必须修改的内容。只清理自己造成的问题。**

编辑现有代码时：
- 不要顺手“改进”相邻代码、注释或格式。
- 不要重构没有问题的代码。
- 遵循现有风格，即使你会用不同方式实现。
- 如果发现无关的死代码，提出来，不要删除。

当你的改动造成孤立代码时：
- 移除因本次改动而不再使用的 import、变量或函数。
- 除非用户要求，不要移除原本就存在的死代码。

检验标准：每一行改动都应能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证目标：
- “添加校验” → “为无效输入编写测试，然后让测试通过”
- “修复 bug” → “编写能复现 bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，给出简短计划：

1. 【步骤】 → 验证：【检查项】
2. 【步骤】 → 验证：【检查项】
3. 【步骤】 → 验证：【检查项】

强成功标准允许独立循环推进。弱标准（例如“让它能用”）需要反复澄清。
